# 공공데이터 확대 세부 계획

## 1. 목표

API 개수를 늘리는 것이 아니라 주민 질문에 답할 수 있는 데이터 묶음을 신뢰성 있게 운영한다. 연계 전 후보 평가, 연계 중 데이터 계약, 공개 전 품질·최신성·법적 검토를 의무화한다.

## 2. 현재 데이터 기준선

| 계층 | 현재 확인 값 | 주의점 |
| --- | ---: | --- |
| 프런트 카탈로그 | 24개 | 실제 수집 여부와 카탈로그 상태가 다를 수 있음 |
| 백엔드 DatasetRegistry | 11개 | 시설 1개와 API 후보 10개 |
| 수집 루틴 | 10개 | 실키·실DB·전체 페이지·금천 필터 검증이 필요 |
| 로컬 API 소스 목록 | 6개 | 실제 백엔드 레지스트리와 개수가 다름 |
| 금천 열린데이터광장 | 화면 표시 159개 | 자동 확인 고유 ID 154개, 분야 편중이 큼 |

신규 수집기 개발 전에 현재 10개 루틴을 “실데이터 운영 가능/부분 구현/소스 미확정/실패”로 재분류한다.

## 3. 데이터 계약

모든 데이터셋은 다음 필드를 가진다.

- 식별자, 주민용 이름, 설명, 담당 도메인
- 원천 기관, 원천 상세 URL, 이용조건·라이선스
- API/파일/관리자 업로드 방식과 인증키
- 원천 공간 범위, 금천 필터 방법, 좌표계, 공간 유형
- 원천 시간 단위, 갱신주기, `observedAt`, `collectedAt`
- 원천 필드, 정규화 필드, 단위, 결측·코드값 규칙
- 예상 건수, 중복키, 페이지네이션, 삭제·변경 감지 방식
- 최신성 SLA, 지연 임계값, 마지막 정상자료 사용 규칙
- 품질 검증, 공개 제한, 개인정보·민감 위치 위험
- 데이터 담당자, 장애 연락, 공개 화면과 시각화 대상

계약이 없거나 원천 상세 URL이 불명확한 데이터는 카탈로그에 “후보”로만 두고 공개 지표에 사용하지 않는다.

2. [민방위대피시설 OA-20044](https://data.geumcheon.go.kr/openinf/openapiview.jsp?infId=OA-20044)
3. [병원 OA-16176](https://data.geumcheon.go.kr/openinf/openapiview.jsp?infId=OA-16176)
4. [약국 OA-16327](https://data.geumcheon.go.kr/openinf/openapiview.jsp?infId=OA-16327)
5. [어린이집 OA-20318](https://data.geumcheon.go.kr/openinf/openapiview.jsp?infId=OA-20318)

대체/후속 후보는 의원 `OA-16231`, 학교 `OA-20520`, 불법주정차 단속 CCTV `OA-20489`다. 1차는 모두 “내 주변”의 검색·목록·상세 행동으로 연결할 수 있어야 한다.

### 2차: 실시간·이동 데이터

- [서울 실시간 통합 API OA-21285](https://data.seoul.go.kr/dataList/OA-21285/A/1/datasetView.do)
- [서울 실시간 인구 API OA-21778](https://data.seoul.go.kr/dataList/OA-21778/A/1/datasetView.do)
- [서울 실시간 상권 API OA-22385](https://data.seoul.go.kr/dataList/OA-22385/F/1/datasetView.do)
- 기상청 단기예보·초단기실황
- 주차 가능면, 전기차 충전 상태, 교통 돌발 등 정확한 금천 범위가 확인된 API

서울 실시간 API는 금천구 전체를 채우는 행정동 통계가 아니라 지정 주요 장소 중심일 수 있다. 가산디지털단지, 독산, 시흥 생활권의 포함 여부와 대표성을 먼저 검증하고 “금천 전체”로 일반화하지 않는다.

### 3차: 분석·행정 데이터

- 동별 연령·1인가구·사업체·시설 접근성
- 예산·집행, 사업 진행, 비식별 민원 분야
- 일자리·채용, 주거·건축, 환경·폐기물

내부 민원·행정 데이터는 개인정보 비식별, 소수집단 노출, 공개 승인, 업데이트 책임이 확보된 뒤 진행한다.

## 5. 수집 아키텍처

```text
원천 API/파일
  -> source adapter
  -> 원천 응답 메타·스키마 검증
  -> staging 또는 수집 배치
  -> 정규화·코드/좌표 변환
  -> 품질 게이트
  -> 버전/스냅샷 저장
  -> 공개 API·카탈로그
  -> 카드/차트/지도
```

### 공통 수집기 규칙

- 데이터셋별 거대한 switch를 확장하기보다 공통 `CollectorAdapter` 계약을 둔다.
- 페이지네이션 상한, 재시도, 지수 지연, 타임아웃, 호출량 제한을 설정으로 둔다.
- 전체 교체는 트랜잭션 또는 새 스냅샷 완료 후 포인터 전환으로 원자성을 보장한다.
- 0건 응답이 정상인지 오류인지 데이터셋 계약으로 판단한다.
- 스키마 변화, 예상 건수 급감·급증, 좌표 범위 이탈 시 공개 반영을 중단한다.
- API 키와 개인정보는 URL·로그·오류·화면에 남기지 않는다.

## 6. 품질 게이트

| 항목 | 기본 규칙 |
| --- | --- |
| 건수 | 이전 정상 대비 ±30% 초과 시 검토, 데이터별 조정 |
| 중복 | 정의된 자연키 중복 0 또는 허용 규칙 명시 |
| 필수값 | 이름·기준시각·공간키 등 계약 필드 충족 |
| 좌표 | 금천 또는 계약 경계 범위, WGS84 변환 검증 |
| 최신성 | SLA 초과 시 `stale`, 두 배 초과 시 경고 |
| 대사 | 원천 `totalCount`와 수집·저장·제외 건수 합치 |
| 스키마 | 필드 삭제·형 변경 시 배치 실패 및 알림 |
| 출처 | 모든 공개 레코드 또는 데이터셋에 원천 링크 보유 |

## 7. 공개 API와 카탈로그

- 공통 응답에 `data`, `meta.status`, `observedAt`, `collectedAt`, `source`, `pagination`을 제공한다.
- 지도 목록 API는 bbox, category, openNow, page, size를 지원한다.
- 대량 원시 데이터를 프런트에서 모두 필터하지 않는다.
- 데이터셋 상세에서 샘플 응답, 필드 정의, 갱신주기, 최근 성공/실패, 다운로드를 제공한다.
- 내부 오류 메시지와 키는 공개 API에 노출하지 않는다.

## 8. 작업 티켓 템플릿

각 신규 데이터는 최소 다음 티켓으로 나눈다.

1. 원천·이용조건·금천 범위 확인
2. 데이터 계약과 샘플 고정
3. 수집 어댑터·정규화 구현
4. 저장 마이그레이션과 품질 검사
5. 공개 API·카탈로그
6. 시각화·빈 값·지연·오류 상태
7. 단위/통합/E2E와 운영 문서

## 9. 완료 정의

- 데이터 계약과 원천 상세 링크가 있다.
- 실제 수집을 반복해 건수·필드·금천 범위를 검증했다.
- 원천 실패나 0건으로 기존 정상 자료가 지워지지 않는다.
- 공개 화면에 기준시각·출처·상태가 보인다.
- 카탈로그·API·차트의 정의와 값이 대사된다.
- 담당자와 장애 대응 절차가 문서화됐다.
