# UI/UX·정보구조 프로젝트 실행계획

- 문서 상태: 실행안 v1.0
- 작성일: 2026-06-18
- 기준 문서: `01-ui-ux-information-architecture-plan.md`
- 적용 범위: `frontend-static/` 공개 서비스와 운영 콘솔의 정보구조 경계
- 계획 기간: 핵심 의사결정 10영업일 + 검증·개발 인계 5영업일

## 1. 실행 목표

이 계획의 목적은 기존 계획의 방향을 실제 업무 티켓과 승인 게이트로 바꾸는 것이다. 15영업일 안에 다음 네 가지를 확정한다.

1. 공개 메뉴를 5개 이하로 줄인 목표 사이트맵과 기존 URL 전환 규칙
2. `오늘의 금천`, `내 주변`, `우리 동` 핵심 3화면의 모바일·데스크톱 사용자 흐름
3. 실데이터·샘플·지연·빈 결과·오류를 구분하는 콘텐츠 및 컴포넌트 계약
4. 주민 과업 테스트를 통과한 고해상도 프로토타입과 프런트엔드 구현 백로그

이 단계의 완료는 운영 코드 전체 개편이 아니라, 구현 중 다시 해석할 필요가 없는 수준의 설계 승인과 개발 인계를 뜻한다.

## 2. 범위와 제외 범위

### 포함

- 공개·운영 라우트 전수 조사와 목표 IA 매핑
- 주민·민원 담당자·데이터 관리자 인터뷰 및 과업 우선순위 확정
- 핵심 3화면의 저해상도/고해상도 와이어프레임
- 공개 내비게이션, 유틸리티 메뉴, 푸터 구조
- 페이지 헤더, 데이터 상태, KPI, 차트, 지도 결과, 필터, 빈 값·오류 상태 명세
- 390px, 768px, 1280px, 1440px 기준 반응형·접근성 명세
- 기존 해시 URL의 리디렉션 또는 호환 정책
- 프런트 안정화·시각화·데이터 확대 작업으로 넘길 구현 백로그

### 제외

- 신규 공공데이터 API 계약·수집기 구현
- 전체 운영 콘솔의 상세 UI 재설계
- 프로덕션 HTML/CSS/JS 전면 구현
- 모든 분야별 화면의 완성형 시각 디자인
- 브랜드 CI 자체 변경

제외 항목은 각각 `03-public-data-expansion-plan.md`, `02-frontend-stabilization-plan.md`, `04-data-visualization-plan.md`에서 이어서 관리한다.

## 3. 현재 기준선

2026-06-18 저장소 기준으로 확인한 구조는 다음과 같다.

| 항목 | 현재 상태 | 설계 영향 |
| --- | --- | --- |
| 라우트 | `main.js`에 12개 라우트 등록 | 통합·이관·호환 정책이 필요 |
| 공개 상단 메뉴 | `index.html`에 10개 메뉴 노출 | 목표 5개 이하와 충돌 |
| 숨은 흐름 | `population`, `geo`는 홈 내부 링크로 진입 | `우리 동`으로 통합 검토 |
| 운영 흐름 | `api`, `api-logs`, `admin`이 공개 SPA에 포함 | 공개/운영 셸 분리 필요 |
| 홈 | 실시간·상권·인구·집계구·카탈로그·API 링크가 한 화면에 집중 | 첫 화면 질문과 행동을 축소해야 함 |
| 데이터 상태 | `local`, `db`, `mixed`와 화면별 문구가 혼재 | 주민용 5상태 계약으로 정규화 필요 |
| 큰 모듈 | `admin.js`, `home.js`, `geo.js`와 `home.css`가 큼 | 디자인 적용 전에 기능 경계 확정 필요 |
| 라우터 | 전체 해시 문자열을 정확히 매칭 | 쿼리·필터 공유 URL을 위한 파싱 보강 필요 |

### 해결해야 할 핵심 문제

- 메뉴 이름이 데이터 종류 중심이라 주민 질문과 바로 연결되지 않는다.
- 홈의 현재 시각과 데이터 기준시각이 시각적으로 혼동될 수 있다.
- 지도 실패 시 목록만으로 과업을 완수하는 경험이 명세되지 않았다.
- `0`, 결과 없음, 미수집, 지연, 오류의 의미가 화면마다 다를 수 있다.
- 기존 링크를 제거하면 북마크와 직접 진입이 깨질 수 있다.

## 4. 목표 정보구조와 URL 전환

### 공개 1차 메뉴

| 순서 | 메뉴 | 대표 질문 | 목표 논리 경로 | 1차 구현 경로 |
| ---: | --- | --- | --- | --- |
| 1 | 오늘의 금천 | 지금 확인할 변화와 주의사항은? | `/today` | `#/home` 유지 |
| 2 | 내 주변 | 내 주변에 이용 가능한 시설은? | `/nearby` | `#/map` 유지 후 별칭 추가 |
| 3 | 우리 동 | 우리 동은 구 평균·다른 동과 무엇이 다른가? | `/dong` | `#/dong` 신규 |
| 4 | 분야별 | 안전·교통·복지 등 한 분야를 더 깊게 보려면? | `/topics/:topic` | `#/topics?topic=` 신규 |
| 5 | 데이터 찾기 | 원자료·API·출처를 어디서 찾나? | `/datasets` | `#/catalog` 유지 |

`이용안내·데이터 정책`은 상단 유틸리티 또는 푸터에 둔다. 모바일에서는 5개 메뉴를 햄버거 안에 같은 순서로 제공하며, 현재 위치와 닫기 버튼의 초점 상태를 보장한다.

### 기존 라우트 전환표

| 현재 경로 | 처리 | 도착 위치 | 비고 |
| --- | --- | --- | --- |
| `#/home` | 유지 | 오늘의 금천 | 메뉴명만 변경 |
| `#/map` | 유지·별칭 | 내 주변 | `#/nearby`도 같은 모듈로 연결 |
| `#/population` | 영구 호환 | `#/dong?section=population` | 직접 링크 보존 |
| `#/geo` | 영구 호환 | `#/dong?section=accessibility` | 비교·접근성 기능 통합 |
| `#/realtime` | 호환 이동 | `#/topics?topic=safety` | 실시간성보다 주민 질문을 우선 |
| `#/indicators` | 호환 이동 | `#/topics` | 분야 허브로 전환 |
| `#/commercial` | 호환 이동 | `#/topics?topic=economy` | 상권 분석 기능 유지 |
| `#/catalog` | 유지 | 데이터 찾기 | 상세 URL 상태 보존 필요 |
| `#/about` | 유지 | 이용안내·정책 | 공개 1차 메뉴에서는 제외 |
| `#/api` | 공개 메뉴 제거 | `/admin/collections` | 인증 운영 콘솔로 이관 |
| `#/api-logs` | 공개 메뉴 제거 | `/admin/logs` | 인증 운영 콘솔로 이관 |
| `#/admin` | 공개 셸에서 분리 | `/admin` | 미인증 접근 차단 |

전환은 두 단계로 진행한다. 1단계에서는 공개 메뉴에서 운영 링크를 제거하고 기존 해시 경로는 호환한다. 2단계에서는 별도 운영 엔트리 또는 서버 경로를 만든 뒤 `/admin` 계열로 이관한다. 정적 서버의 fallback과 인증 동작이 확정되기 전에는 기존 경로를 삭제하지 않는다.

## 5. 화면 설계 명세

### 오늘의 금천

첫 화면은 `위치와 기준시각 → 주의 알림 → 4개 요약 → 3개 행동` 순서로 고정한다.

| 영역 | 필수 내용 | 상호작용 | 예외 상태 |
| --- | --- | --- | --- |
| 페이지 헤더 | 선택 위치, 데이터 상태, 실제 기준시각 | 관심 동 변경 | 위치 미선택, 갱신 지연 |
| 주의 알림 | 위험/주의/정보 등급, 근거, 행동 | 세부 안내·문의 이동 | 알림 없음과 조회 실패 구분 |
| 4개 요약 | 날씨·대기, 안전, 교통, 생활서비스 | 상세 분야 이동 | 값 없음, 샘플, 부분 오류 |
| 빠른 행동 | 내 주변, 우리 동, 민원·문의 | 명확한 링크/버튼 | 외부 링크 실패 안내 |
| 변화 요약 | 이전 기준 대비 3개 변화 | 정의·원자료 열기 | 비교 기간 부족 |
| 지도 미리보기 | 핵심 생활시설과 목록 | 내 주변으로 이동 | 타일 실패 시 목록 유지 |
| 출처 요약 | 제공기관, 기준일, 품질 상태 | 데이터 상세 이동 | 복수 출처 개별 표시 |

실행 시간 시계, 광고형 대형 그라데이션 배너, API 내부 상태 바로가기는 제거 대상으로 표시한다.

### 내 주변

| 뷰포트 | 레이아웃 | 핵심 규칙 |
| --- | --- | --- |
| 1280px 이상 | 필터 + 목록 + 지도 분할 | 목록과 지도 선택 상태 동기화 |
| 768px | 상단 필터 + 목록/지도 탭 | 마지막 선택 탭 유지 |
| 390px | 검색 → 필터 시트 → 결과 카드 | 지도는 별도 탭, 결과 수 먼저 안내 |

URL에는 주소 식별자 또는 좌표, 거리, 시설 유형, 운영 중 여부, 선택 항목, 지도 범위를 직렬화한다. 개인정보가 될 수 있는 정밀 현재 위치는 동의 없이 저장하거나 공유 URL에 넣지 않는다. 지도 타일·지도 라이브러리 실패 시에도 검색, 필터, 전화, 주소, 길찾기 링크를 사용할 수 있어야 한다.

### 우리 동

화면 순서는 `동 선택 → 핵심 요약 → 추세 → 구성 → 시설 접근성 → 원자료`로 고정한다.

- 기본 비교는 구 평균과 이전 기간이다.
- 다른 동 비교는 사용자가 명시적으로 선택한다.
- 지표마다 값, 단위, 기간, 분모, 출처, 갱신주기를 표시한다.
- 표본 또는 비교 대상이 부족하면 순위와 우열 표현을 사용하지 않는다.
- 갱신주기가 다른 데이터는 같은 축이나 하나의 종합점수로 합치지 않는다.
- 기존 `population`과 `geo`의 기능 목록을 작성해 유지·통합·보류를 하나씩 결정한다.

### 분야별과 데이터 찾기

분야별은 분야명보다 주민 질문 3~5개를 먼저 보여준다. 질문마다 대표 지표 1개, 원인을 보는 진단 지표, 관련 공공 서비스 행동을 연결한다. 데이터 찾기는 검색창을 첫 초점 가능 요소로 두고 분야, 공간 유형, 갱신주기, 제공 형식, 제공기관, 공개 상태 필터를 제공한다.

## 6. 공통 콘텐츠·컴포넌트 계약

### 데이터 상태

| 상태 | 주민 표시 | 필수 메타데이터 | 금지 표현 |
| --- | --- | --- | --- |
| `live` | 최신 데이터 | 관측시각, 수집시각, 출처 | 근거 없는 `실시간` |
| `sample` | 샘플 데이터 | 샘플 범위, 생성일, 목적 | `LIVE`, 실제 현황 암시 |
| `stale` | 갱신 지연 | 마지막 정상 시각, 지연 기준, 대체 여부 | 현재값처럼 무표시 |
| `empty` | 해당 조건 결과 없음 | 검색 조건, 초기화 행동 | `0건`과 혼용 |
| `error` | 데이터를 불러오지 못함 | 영향 범위, 대체 자료, 재시도 | 허위 `0`, 내부 예외 전문 |

`observedAt`은 원천 데이터의 기준시각, `collectedAt`은 플랫폼이 수집한 시각으로 정의한다. 현재 브라우저 시각은 어느 것도 대신할 수 없다.

### 컴포넌트 완료 조건

| 컴포넌트 | 필수 변형 | 접근성·행동 기준 |
| --- | --- | --- |
| 데이터 상태 배지 | live/sample/stale/empty/error | 텍스트+아이콘, 도움말 연결 |
| KPI 카드 | 정상/증감 없음/비교 불가/오류 | 단위·기간·분모·비교 기준 제공 |
| 차트 카드 | 로딩/정상/빈 값/오류 | 요약 문장, 표 보기, 다운로드 |
| 지도 결과 카드 | 운영/종료/정보 없음 | 전화·주소·길찾기 키보드 접근 |
| 필터 바 | 기본/활성/접힘 | 활성 필터 요약, 전체 초기화 |
| 페이지 헤더 | 위치/범위/기준일 | 화면에서 얻는 답을 한 문장으로 설명 |
| 상태 메시지 | 성공/진행/실패 | 필요한 경우 `aria-live` 사용 |

디자인 토큰은 기존 `css/tokens.css`를 감사한 뒤 의미 기반 이름으로 정리한다. 페이지 CSS에서 임의 색상·간격을 추가하기 전에 토큰 사용 가능 여부를 먼저 확인한다.

## 9. 사용자 조사와 검증 계획

### 참여자

| 집단 | 인원 | 확인할 내용 |
| --- | ---: | --- |
| 주민·보호자 | 5~8명 | 모바일 탐색, 시설 찾기, 용어 이해 |
| 민원 담당자 | 2명 | 빈번한 질문, 문의·신청 연결 |
| 데이터 관리자 | 1명 | 출처·갱신·오류 상태와 운영 경계 |

가능하면 60대 이상 2명 이상, 모바일 주사용자 3명 이상, 외국인 주민 또는 쉬운 한국어 요구를 대변할 참여자 1명을 포함한다. 개인정보가 포함된 원문 민원은 사용하지 않고 유형과 빈도만 비식별 집계한다.

### 테스트 과업

1. 오늘 금천에서 주의할 내용을 찾고 데이터 기준시각을 말한다.
2. 가상의 주소 주변에서 현재 이용 가능한 약국을 찾고 전화 또는 길찾기를 선택한다.
3. 독산1동을 구 평균 및 이전 기간과 비교해 차이 한 가지를 설명한다.
4. 한 지표의 출처·단위·원자료를 찾는다.
5. 데이터가 지연되거나 샘플일 때 그 의미를 설명한다.

### 측정과 판정

- 과업 성공률: 성공한 과업 수 / 전체 시도 수, 목표 80% 이상
- 첫 클릭 성공률: 올바른 첫 진입을 선택한 비율, 목표 70% 이상
- 주의 정보 확인 시간: 중앙값 20초 이하
- 기준일·출처 발견률: 90% 이상
- 심각도 1 문제: 과업 중단 또는 잘못된 행동 유발, 승인 전 0건
- 심각도 2 문제: 큰 우회·반복이 필요, 담당과 수정 기한 필수

소규모 표본의 비율은 방향 확인용이며 통계적 대표성으로 해석하지 않는다. 결과 문서에는 관찰 사실, 해석, 설계 결정, 남은 불확실성을 구분한다.

## 12. 단계 게이트

### G1 — 조사·IA 승인(D5)

- 전체 라우트, 진입 링크, 핵심 질문, 데이터 출처가 매핑되었다.
- 공개 메뉴가 5개 이하이고 운영 메뉴가 분리되었다.
- 기존 URL별 유지·호환·이관 결정이 있다.
- 상위 과업과 제외 과업에 조사 또는 운영 근거가 있다.

### G2 — 저해상도 설계 승인(D10)

- 핵심 3화면의 모바일·데스크톱 흐름이 있다.
- 정상, 샘플, 지연, 빈 값, 오류 상태가 포함되어 있다.
- 지도 없이 내 주변 과업을 완료할 수 있다.
- 과업 성공률 80%, 첫 클릭 성공률 70% 이상이며 심각도 1 문제가 없다.

### G3 — 고해상도·개발 인계 승인(D15)

- 실제 한국어 길이, 단위, 큰 수, 긴 시설명이 검증되었다.
- 390/768/1280/1440px와 200% 확대 명세가 있다.
- 모든 컴포넌트와 화면 영역이 구현 티켓에 연결되었다.
- 데이터 미확정 항목은 샘플 상태로 명시되며 공개 전 조건이 기록되었다.
- 제품, 현업, UX, FE, 데이터, QA 리뷰의 미해결 차단 이슈가 없다.

## 13. 개발 백로그 분할 원칙

설계 인계 후 구현 PR은 다음 순서로 분리한다.

1. 테스트 기준선과 라우트 호환 장치
2. 공개 내비게이션 5개 개편과 운영 링크 제거
3. 공통 데이터 상태 모델·배지·빈 값·오류 컴포넌트
4. 오늘의 금천 구조 개편
5. 내 주변 목록 우선·지도 실패 대응·URL 상태
6. 우리 동 통합 라우트와 기존 `population`·`geo` 호환
7. 분야별 허브와 상권·실시간·지표 이관
8. 데이터 찾기 검색·필터·상세 메타데이터
9. 운영 콘솔 별도 엔트리·인증 경계
10. 반응형·접근성·시각 회귀 보정

각 티켓은 사용자 이야기, 대상 경로, 데이터 상태, 수용 기준, 접근성 조건, 테스트, 선행 조건을 가져야 한다. 기능 변경과 대규모 리팩터링, 데이터 정의 변경을 한 PR에 동시에 넣지 않는다.

## 16. 최종 완료 정의

- 공개 IA와 운영 IA가 분리되고 공개 1차 메뉴가 5개 이하다.
- 기존 12개 라우트 모두에 유지·통합·호환·이관 결정이 있다.
- 핵심 3화면은 모바일과 데스크톱에서 하나의 질문과 하나의 주 행동을 가진다.
- 주민이 20초 안에 주의 정보, 상태, 기준시각, 다음 행동을 확인할 수 있다.
- 3번 이내 조작으로 시설 찾기와 동 비교에 도달한다.
- 지도 없이도 내 주변 검색, 결과 확인, 연락·길찾기가 가능하다.
- 데이터 상태 5종과 관측시각·수집시각·출처가 공통 계약으로 확정되었다.
- 핵심 과업 테스트, 접근성 검토, 개발 가능성 리뷰를 통과했다.
- 모든 승인된 화면·컴포넌트가 우선순위와 테스트가 있는 개발 티켓으로 인계되었다.

## 17. 즉시 착수할 첫 3일

1. D1: `index.html`, `main.js`, 페이지 내부 링크를 기준으로 라우트·진입 링크 인벤토리를 만든다.
2. D1: 제품 책임자, UX, 프런트엔드, 데이터, QA 승인자를 지정하고 15일 캘린더를 잠근다.
3. D1~D2: 주민 5~8명, 민원 담당자 2명, 데이터 관리자 1명 모집을 시작한다.
4. D2: 각 화면을 `사용자 질문 / 주 행동 / 데이터 출처 / 현재 문제 / 목표 위치`로 분류한다.
5. D2: 공개·운영 내비게이션의 1단계 분리안을 검토한다.
6. D3: 인터뷰를 시작하고, 동시에 목표 사이트맵과 기존 URL 전환안 1차본을 만든다.
