# 03 공공데이터 확대 진행 기록

기준일: 2026-06-22

## 2026-06-22 일반 인터넷 재검증

- 일반 Wi-Fi 연결에서 기본 HTTPS(`example.com:443`)는 정상 연결되었다.
- 서울시 OpenAPI DNS는 기존 방화벽 허용 대상과 같은 `115.84.165.45`로 확인했다.
- 방화벽에 등록된 중계기 전용 Python 3.13으로 `openapi.seoul.go.kr:8088`을 재검증했으나 TCP 연결이 계속 시간 초과되었다.
- 따라서 아이폰 핫스팟만의 차단 문제라는 가설은 기각한다. 현재 사용 가능한 두 회선 모두에서 서울시 비표준 포트 경로를 사용할 수 없다.
- 기존 1,644행 성공 스냅샷은 삭제하거나 0행으로 대체하지 않고 유지한다. 이 수치는 과거 수집 행이며 현재 실시간 응답으로 해석하지 않는다.
- 다음 결정은 기존 외부 서버 사용, 신규 외부 중계 서버 마련, 또는 스냅샷 유지 중 하나다. 외부 서버 선택과 비용 발생은 사용자 승인 전까지 보류한다.

## 2026-06-22 운영 기준 확정

- 공공 Wi-Fi는 기존 마지막 성공 스냅샷 1,644행을 사용한다.
- 서울시 HTTP 8088 실시간 수집은 기본 비활성화하며 외부 서버를 도입하지 않는다.
- `WIFI_REALTIME_COLLECTION_ENABLED=false`이면 DB 실행 스크립트가 Wi-Fi 중계기를 시작하지 않고, 런타임 토큰과 Wi-Fi 예약 수집도 비활성화한다.
- 일반 HTTPS 수집기와 로컬 백엔드 실행은 이 결정과 독립적으로 유지한다.
- 화면에서 공공 Wi-Fi를 선택하면 `마지막 성공 데이터`, 1,644행, 기준일, 실시간 수집 비활성화 상태를 함께 표시한다. 화면의 금천구 시설 수와 전체 보존 스냅샷 행 수는 같은 의미로 비교하지 않는다.
- 접속 가능한 회선이 확보되면 `WIFI_REALTIME_COLLECTION_ENABLED=true`와 필요한 키를 명시적으로 설정한 뒤 Wi-Fi 수집만 재활성화한다.

## 현재 상태

03 단계의 정적 감사, 임시 운영정책, 공통 품질 게이트, 공개 데이터 계약 API와 핵심 실데이터 연결을 구현했다. 상가·인구·에어코리아는 운영 DB 반복 수집을 통과했고 CCTV는 관리자 CSV 교체 경로로 935건을 검증했다. 주차장은 공식 시설 25건을 `parking-lots`, 기존 서울 공간행 131건을 `parking-spaces`로 분리해 서로 다른 입자도의 기준선 비교를 해소했다.

## 이번 반영

- 서울 열린데이터 수집 URL 다섯 곳을 HTTP에서 HTTPS로 변경했다.
- CCTV·주차장·공공 와이파이 레지스트리의 출처를 계획에 명시된 금천구 공식 상세 URL로 수정했다.
- 레지스트리의 과도한 `API 가능` 상태를 `부분 구현·실키 검증 필요` 또는 `소스 미확정`으로 재분류했다.
- 서울 수집기 HTTPS·키 마스킹과 레지스트리 분류·공식 링크 회귀 테스트를 추가했다.
- 0건 응답을 성공 처리하지 않고 저장 호출 전에 실패시켜 기존 정상 스냅샷을 보존한다.
- 소스 미확정 세 종은 공개 대상과 자동 수집에서 제외하고, 수동 동기화도 외부 호출 없이 `skipped`로 처리한다.
- 상가·대기·시설 데이터의 유효 행 비율이 70% 미만이면 삭제 직전 품질 게이트에서 중단한다.
- 현재 레지스트리가 표현하는 HTTPS 상세 URL·공간 유형·환경변수·필수 필드·대상 화면을 자동 감사한다.
- 데이터셋별 자연키, 초기 정상 건수 범위, ±30% 변동 예산, 최신성 SLA, 마지막 정상자료 보존 기간을 별도 운영 정책 레지스트리로 분리했다.
- 마지막 성공 수집의 원천 건수를 조회해 다음 수집이 ±30%를 벗어나면 저장 전에 중단한다.
- 공개 데이터 계약 조회 `GET /api/public/datasets/contracts`를 추가했다. 공개 데이터셋만 대상으로 원천 링크, 필드, 자연키, 정상 건수 범위, SLA, 보존기간, 승인·검토 상태와 역할 담당자를 제공한다.
- 공개 계약 응답에서는 API 키 값과 키 환경변수명 같은 내부 설정을 제외한다.

## 정책 결정 기록

2026-06-19 사용자 승인에 따라 다음 권장안을 적용했다.

- 활성 수집기 7종과 관리자 시설 업로드의 기술 기준을 `PROVISIONALLY_APPROVED`로 설정한다.
- 임시 승인 유효기간은 2026-06-19부터 2026-09-19까지 3개월이다.
- 2026-09-19까지 실수집 건수·지연 이력·오탐을 검토해 범위와 SLA를 재승인한다.
- 2026-09-19부터 재승인 전까지 신규 스냅샷 교체를 품질 게이트에서 차단하고 마지막 정상자료를 유지한다.
- 이용조건은 데이터별 원문 확인 전까지 `REVIEW_REQUIRED`로 유지한다.
- 담당자는 개인이 아니라 `데이터정책 담당부서`, `플랫폼 운영 담당`, `정보보안·개인정보 담당` 역할로 지정한다.
- 소스 미확정 3종은 `DRAFT`이며 비공개·수집 제외 상태를 유지한다.
- 같은 날 추가 사용자 승인에 따라 서울 OpenAPI의 평문 HTTP 8088 예외를 허용하지 않는다.
- 당시 air-quality, bike-stations, cctv-stations, parking-lots, public-wifi는 기술 계약은 유지하되 `collectionEnabled=false`로 중단했다.
- 수동·전체·예약 동기화 모두 외부 요청 전에 `skipped` 처리하며 공식 HTTPS 대체 원천이 검증될 때만 재활성화한다.
- 데이터 연결 검증 필요성을 재검토한 뒤 `COLLECTOR_ALLOW_INSECURE_SEOUL_HTTP=true`인 명시적 로컬 프로세스에서만 HTTP를 허용했다.
- 기본값은 `false`이며 자동 스케줄러와 운영 수집 정책은 계속 차단한다. 로컬 예외에서도 키·요청 URL 로그 마스킹, 정상 건수 범위, 기존 스냅샷 보호는 유지한다.

2026-06-20 사용자 승인에 따라 공간 범위 정책을 다음과 같이 확정했다.

- 범위 밖 행을 삭제하지 않고 모든 시설·상가 행을 `GEUMCHEON`, `BORDER_AREA`, `EXTERNAL_REFERENCE` 중 하나로 분류한다.
- 공식 통계와 일반 공개 조회의 기본 범위는 `GEUMCHEON`만 사용한다.
- 상권분석 화면의 기본값은 **금천구만**이며, 사용자가 선택한 경우에만 `GEUMCHEON+BORDER_AREA`를 함께 조회한다.
- `EXTERNAL_REFERENCE`는 대사·원천 추적용으로 보존하되 공식 통계와 상권분석에는 포함하지 않는다.
- 공식 원천의 `district=금천구` 또는 경계 도형 내부 좌표는 `GEUMCHEON`으로 우선 판정한다. 주소에 금천구가 명시된 행도 `GEUMCHEON`, 인접 생활권은 `BORDER_AREA`, 그 밖의 참조 행은 `EXTERNAL_REFERENCE`로 분류한다.
- 현재 경계 도형은 프런트 동 경계를 합친 간이 경계(`quality=SIMPLIFIED`)다. 공식 행정경계로 교체해도 분류값과 공개 계약은 그대로 유지한다.

같은 날 주차장 전환 권장안을 승인해 다음 정책을 적용했다.

- 공공데이터포털의 25개 시설을 공식 `parking-lots` 기준선으로 사용한다.
- 구 서울 원천 131개 공간행과 성공 수집 이력은 `parking-spaces` 참고 데이터셋으로 이동하며 삭제하지 않는다.
- `parking-spaces`는 `PARKING_SPACE_REFERENCE` 범주로 저장하고 공식 통계와 기본 시설 조회에서 제외한다.
- 두 데이터셋의 품질 게이트는 독립 기준선을 사용하며 25와 131을 증감률로 비교하지 않는다.

- 필드 정의, 자연키, 초기 SLA와 원천 상세 URL을 데이터셋 레지스트리·운영 정책에 등록했다. 등록 데이터셋은 12종에서 17종으로 늘었다.
- 실제 API가 평문 HTTP 8088만 안내하므로 기존 HTTPS 정책과 Wi-Fi 전용 중계 목적지 제한을 위반하지 않도록 5종 모두 `DRAFT`, 수집 비활성, 공개 비노출, `SCREEN_ONLY`로 유지했다.
- 백엔드 전체 회귀는 81건 모두 통과했다.

다음 구현을 위해서는 기존 Wi-Fi 중계기의 목적지를 넓히는 것이 아니라, 생활시설 5종 전용 격리 중계기를 추가할지 또는 운영자가 공식 파일을 정기 업로드하는 경로를 사용할지 결정해야 한다.

## 2026-06-21 생활시설 전용 중계 구현 완료

- Wi-Fi와 분리된 Python 프로세스, 전용 포트 `127.0.0.1:18089`, 별도 런타임 토큰을 적용했다.
- 상류 목적지는 `openapi.seoul.go.kr:8088`, 서비스 허용목록은 5개로 고정했다. 다른 서비스 ID는 호출 전에 거부한다.
- `run-backend-db.ps1`이 Wi-Fi 중계기와 생활시설 중계기의 준비 상태를 순서대로 확인한 뒤 최종 JAR를 실행한다. 백엔드 종료·시작 실패·준비 실패 때 현재 실행이 소유한 중계기만 정리한다.
- 필수 필드, 자연키 중복, WGS84 좌표 쌍·범위, 유효 비율, 행 수, 이전 성공 대비 건수 급변을 중계기와 백엔드 양쪽에서 검증한다. 좌표가 없는 공식 행은 추정하지 않는다.
- 생활지도에 복지·대피시설·병원·약국·어린이집 필터와 출처·기준일 표시를 연결했다. 이용조건은 계속 `SCREEN_ONLY`다.
- 백엔드 전체 81건, 최종 변경 집중 회귀 23건, 프런트엔드 41건, Python 3건, PowerShell 생명주기 3건이 통과했다.
- 최종 JAR 교체 후 Spring Boot, Wi-Fi 중계기, 생활시설 중계기를 통합 재기동했고 `/actuator/health`는 `UP`이다.
- 실제 자동·개별 수집에서 5종 모두 약 18초 후 안전한 `TIMEOUT`으로 종료됐다. 신규 정상 스냅샷은 없으며 기존 자료를 삭제하거나 0건으로 교체하지 않았다.

현재 코드·JAR·운영 문서는 완료됐다. 실데이터 최초 기준선 확정만 현재 서버의 서울시 TCP 8088 통신이 가능해진 뒤 남아 있다. 다음 외부 판단 지점은 서버/방화벽에서 8088 아웃바운드를 허용할지, 허용할 수 없다면 별도 외부 중계 서버를 선택할지다.

### 8088 네트워크 진단

- Windows 방화벽 기본 아웃바운드 정책은 허용이며 WinHTTP 프록시는 없었다.
- 관리자 권한으로 중계기 Python→서울시 DNS 주소 `115.84.165.45`→TCP 8088만 허용하는 규칙을 적용했다.
- 적용 뒤에도 실제 Python TCP 연결은 시간 초과했다.
- `data.seoul.go.kr:443`, `data.go.kr:443`, 일반 HTTPS는 성공하고 `openapi.seoul.go.kr:443/8088`만 실패했다.
- Google·Cloudflare 공용 DNS도 동일 IP를 반환했다. 당시 네트워크는 iPhone 핫스팟이므로 다음 검증은 유선 또는 일반 광대역망에서 수행해야 한다.
