# 실행·품질·운영 세부 계획

## 1. 목표

큰 개편을 한 번에 배포하지 않고 기능 플래그와 단계별 품질 게이트로 위험을 줄인다. 코드 완료뿐 아니라 데이터 운영, 접근성, 성능, 장애 대응, 담당자 인수까지 완료 상태에 포함한다.

## 2. 작업 운영 방식

### 스프린트

- 2주 스프린트, 주 1회 화면·데이터 데모
- 각 스프린트는 주민 과업 1개와 기술 안정화 1개를 함께 완료
- 백로그는 `UX`, `FE`, `DATA`, `VIZ`, `OPS` 접두어 사용
- 작업 크기는 개발·테스트·문서 포함 최대 2~3일을 목표로 분할

### 의사결정자

| 역할 | 책임 |
| --- | --- |
| 제품 책임자 | 주민 우선순위, 공개 범위, 출시 승인 |
| UX 담당 | 사용자 조사, IA, 프로토타입, 접근성 설계 |
| 프런트 담당 | 컴포넌트, 상태, 반응형, 자동화 |
| 백엔드/데이터 담당 | 데이터 계약, 수집, 저장, 공개 API |
| 데이터 담당 부서 | 원천 정의, 갱신·품질, 장애 확인 |
| QA/접근성 | 테스트 매트릭스, 회귀, 접근성 검증 |
| 운영 담당 | 모니터링, 키, 배포·롤백, 인수 |

## 3. 릴리스 전략

### 환경

- 로컬: 샘플/Mock을 명시적으로 사용
- 개발: 고정된 테스트 데이터와 외부 API 모킹
- 스테이징: 운영과 같은 설정, 비공개 실데이터 검증
- 운영: 승인된 데이터셋과 기능 플래그만 공개

### 단계 배포

1. 내부 직원만 새 IA 사용
2. 홈 일부 사용자 또는 별도 베타 URL
3. 내 주변·우리 동 순차 공개
4. 전체 공개 후 기존 화면 2주 병행
5. 지표 확인 후 기존 화면 종료

데이터셋도 화면과 별도 플래그를 사용해 수집 성공이 곧 공개를 의미하지 않게 한다.

## 4. 품질 게이트

### 설계 게이트

- 사용자 질문, 주 행동, 데이터 정의, 정상/빈 값/오류 상태 승인
- 모바일과 키보드 흐름 포함
- 참고 사례와 구조적 유사성 검토

### 코드 게이트

- 관련 단위·통합·E2E 통과
- 신규 콘솔 오류, 접근성 중대 오류 없음
- 보안 문자열 처리와 운영/공개 경계 검토
- 성능 예산 변화 기록

### 데이터 게이트

- 실제 수집 2회 이상 성공
- 원천 건수와 저장·제외 건수 대사
- 스키마·좌표·중복·최신성 검사 통과
- 원천 장애 시 기존 정상자료 보존 검증
- 출처·라이선스·담당자 등록

### 출시 게이트

- 390/768/1280/1440px 핵심 과업 통과
- Chrome/Edge 및 모바일 브라우저 표본 검증
- 스크린리더·키보드·200% 확대 점검
- 외부 API, 지도 타일, CDN, 백엔드 장애 시나리오 통과
- 모니터링, 공지, 롤백, 연락망 확인

## 5. 관측성과 알림

### 서비스 지표

- 화면 가용성, API 성공률, p95 응답시간
- JS 오류율, 실패 요청, 핵심 과업 이탈
- LCP/CLS/INP 등 실제 사용자 성능

### 데이터 지표

- 데이터셋별 마지막 성공, 마지막 정상 기준시각, 지연 시간
- 원천/수집/저장/제외 건수
- 스키마 오류, 좌표 이탈, 중복, 예상 건수 편차
- 연속 실패 횟수와 마지막 오류 범주

### 알림 단계

- Warning: 최신성 SLA 초과 또는 1회 실패
- Critical: 두 배 SLA 초과, 연속 3회 실패, 건수 급감, 공개 API 오류
- Security: 키 노출 가능성, 인증 실패 급증, 관리자 이상 접근

주민 화면은 내부 스택·키·URL을 보여주지 않고 “몇 시 자료를 표시 중인지”와 영향만 안내한다.

## 6. 장애 시나리오

| 장애 | 주민 화면 | 운영 대응 |
| --- | --- | --- |
| 외부 API 지연 | 마지막 정상자료 + 지연 배지 | 재시도, 제공기관 상태 확인 |
| 0건 비정상 응답 | 0으로 공개하지 않고 갱신 실패 | 품질 게이트 차단 |
| 지도 타일 실패 | 목록·검색 유지, 대체 배경 | 타일 제공처·네트워크 확인 |
| 일부 API 실패 | 실패 영역만 안내 | 데이터셋별 알림 |
| 스키마 변경 | 이전 정상자료 유지 | 계약 수정·어댑터 배포 |
| 프런트 신규 버그 | 기능 플래그 해제 또는 이전 버전 | 오류 증거 보존·롤백 |
| 관리자 인증 문제 | 공개 서비스 영향 없음 | 별도 운영 경로 복구 |

## 7. 테스트 체계

- 단위: 계산, 포맷, 상태 변환, 데이터 매핑
- 계약: 외부 API 샘플과 스키마
- 통합: 수집→DB→공개 API
- E2E: 주민 핵심 과업, 운영 핵심 과업
- 시각 회귀: 홈·지도·동 비교의 주요 상태와 폭
- 접근성: 자동 axe + 수동 키보드/스크린리더
- 성능: Lighthouse CI 또는 동등 도구 + 실제 사용자 지표
- 복구: 외부 API·DB·타일 장애 주입

## 8. 문서와 인수

출시 전 다음 문서를 최신화한다.

- 데이터셋 계약과 담당자 목록
- 환경변수·키 발급·회전 절차
- 수집 실패와 수동 재시도 절차
- 배포·롤백·DB 마이그레이션 절차
- 공개 장애 공지 템플릿
- 화면별 지표 정의와 민원 응대 FAQ
- 신규 담당자 30분 운영 점검 체크리스트

## 9. 위험 관리

| 위험 | 완화 |
| --- | --- |
| 범위가 계속 늘어남 | 주민 과업 점수와 단계 게이트로 1차 범위 고정 |
| 실제 API 품질이 낮음 | 연계 전 샘플·건수·스키마 검증, 마지막 정상자료 보존 |
| UI 개편과 버그 수정 충돌 | 안정화와 시각 변경 PR 분리 |
| 서초 모방 인상 | 질문 중심 IA, 금천 모티프, 구조 유사성 리뷰 |
| 데이터가 많아 화면 과밀 | 요약→진단→상세 계층과 공개 메뉴 5개 제한 |
| 운영 담당 부재 | 데이터 계약에 담당자 필수, 미지정 데이터 공개 금지 |
| 네트워크 제한으로 검증 지연 | 의존성 캐시·CI·고정 샘플과 실환경 검증을 분리 |

## 10. 출시 후 30일

- 1주차: 오류·성능·데이터 최신성 매일 검토
- 2주차: 검색어, 빈 결과, 이탈 과업 분석
- 3주차: 주민·민원 담당자 인터뷰와 P1/P2 개선
- 4주차: KPI 리뷰, 다음 데이터 웨이브 승인

기능 수가 아니라 과업 성공률, 데이터 신뢰, 장애 복구 시간을 기준으로 다음 투자를 결정한다.

## 11. 최종 완료 정의

- 사용자·설계·코드·데이터·출시 게이트가 모두 승인됐다.
- 운영 모니터링과 담당자 인수가 완료됐다.
- 롤백과 데이터 지연 안내가 실제로 검증됐다.
- 문서와 화면의 출처·정의·기준시각이 일치한다.
- 공개 후 30일 검토와 다음 단계 결정 일정이 잡혀 있다.
