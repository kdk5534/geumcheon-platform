# 금천구 데이터플랫폼 전면 개선 마스터 계획

- 문서 상태: 초안 v1.0
- 작성일: 2026-06-18
- 적용 범위: `frontend-static/`, `backend-egovframe-skeleton/`, `database/`, 데이터 수집·운영 문서
- 목표 사용자: 금천구 주민·민원인 우선, 소상공인·정책 담당자·플랫폼 관리자 보조

## 1. 목적

금천구의 생활·안전·교통·복지·보건·환경·상권 정보를 민원인이 한눈에 이해하고 다음 행동까지 할 수 있는 공공 데이터 서비스로 개선한다. 단순히 데이터와 차트를 늘리는 것이 아니라 다음 세 문제를 함께 해결한다.

1. 공공 화면의 정보구조와 시각적 완성도를 주민 중심으로 재설계한다.
2. 프런트엔드의 많은 오류를 재현·분류·자동 검증할 수 있는 안정화 체계를 만든다.
3. 공공데이터 연계를 신뢰 가능한 수집·품질·출처 체계 위에서 단계적으로 확대한다.

이 문서는 큰 방향과 순서를 정한다. 실제 실행 단위와 완료 기준은 `docs/renewal/`의 세부 계획을 따른다.

## 2. 이번 조사에서 확인한 현재 기준선

### 구현 현황

- 공개 프런트는 빌드 없는 정적 SPA이며 12개 라우트 모듈을 가진다. 상단 공개 메뉴에는 상황판, 실시간, 지표, 지도, 상권, 카탈로그뿐 아니라 API 상태·로그·관리자까지 함께 노출된다.
- 프런트 데이터 카탈로그에는 24개 항목, 백엔드 `DatasetRegistry`에는 11개 데이터셋, `PublicDataCollectorService`에는 10개 수집 루틴이 있다.
- 로컬 샘플은 시설 30건, 인구 3개 동, 지표 4개 등으로 실제 서비스 범위보다 작다.
- 프런트 JS 문법 검사는 통과했지만 프런트 단위·통합·E2E·접근성·시각 회귀 테스트 구성이 없다.
- `home.js` 903행, `admin.js` 997행, `geo.js` 771행, `home.css` 1,472행 등 핵심 파일이 커져 변경 영향 추적이 어렵다.
- 실제 API와 로컬 Mock을 자동 병합하지만, 현재 시각을 데이터 기준시각처럼 보이게 하거나 Mock과 실데이터가 한 화면에서 섞일 수 있는 신뢰도 위험이 있다.
- 화면 스크린샷에서 지도 타일 실패, 빈 값과 0의 혼용, 첫 화면의 높은 정보밀도, 긴 상단 메뉴, 지도 중심 3열 레이아웃의 과밀을 확인했다.
- 백엔드는 이번 환경에서 Maven Central 네트워크 제한으로 기동하지 못했다. 정적 서버는 정상 응답했다. 이 제한은 구현 결함과 분리해 기록한다.

### 데이터 조사 현황

- 2026-06-18 기준 금천구 열린데이터광장은 목록에 159건을 표시했다. 페이지 자동 수집으로 고유 ID 154건을 확인했으며 보건 56, 문화관광 40, 산업경제 20, 교육 15, 환경 10, 일반행정 5, 안전 3, 복지 2, 교통 2, 도시관리 1건이었다.
- 주민 활용 가치가 높은 정확한 후보로 공공와이파이 `OA-20906`, 사회복지시설 `OA-20394`, 어린이집 `OA-20318`, 민방위대피시설 `OA-20044`, 병원 `OA-16176`, 의원 `OA-16231`, 약국 `OA-16327`, 불법주정차 CCTV `OA-20489`, 학교 `OA-20520`, 주차장 `OA-12770`, CCTV `OA-12253`을 확인했다.
- 서울 실시간 도시데이터는 실시간 통합 API `OA-21285`, 실시간 인구 `OA-21778`, 실시간 상권 `OA-22385`를 공개한다. 금천구 관심 지점의 실제 포함 범위는 연계 착수 전에 별도 검증해야 한다.

## 3. 제품 원칙

### 주민 질문이 메뉴보다 먼저다

첫 화면은 “데이터가 무엇이 있는가”보다 다음 질문에 즉시 답해야 한다.

- 지금 금천에서 주의할 일은 무엇인가?
- 내 주변에서 이용할 수 있는 시설은 어디인가?
- 우리 동은 다른 동과 무엇이 다른가?
- 이 수치는 언제, 어디에서 왔으며 믿어도 되는가?
- 문제가 있으면 어디로 이동하거나 문의해야 하는가?

### 0, 없음, 미수집, 오류를 구분한다

`0건`, `해당 없음`, `아직 수집하지 않음`, `갱신 지연`, `연결 실패`는 서로 다른 상태다. 모든 KPI와 차트에 이 구분을 적용한다.

### 첫 화면은 요약, 다음 화면은 원인과 행동

기본 화면은 핵심 상태와 변화만 보여준다. 추세, 동별 비교, 세부 행, 다운로드는 단계적으로 펼친다. 한 화면에 가능한 모든 차트를 쌓지 않는다.

### 지도와 차트는 표와 설명을 동반한다

지도 실패 시 목록으로 같은 정보를 이용할 수 있어야 하며, 차트는 데이터 표·단위·기준일·출처를 제공해야 한다.

### 실데이터 확대보다 신뢰 계약이 먼저다

새 API마다 수집 주기, 필드, 공간 범위, 실패 처리, 최신성 임계값, 라이선스, 담당자를 등록한 뒤 공개한다.

### 서초의 구조가 아닌 금천의 사용 맥락을 설계한다

서초 플랫폼의 중앙 지도와 3열 카드 배치를 복제하지 않는다. 금천은 G밸리의 업무 이동, 안양천 생활권, 동별 주거·산업 혼합, 주민 생활 서비스 탐색을 중심으로 고유한 “생활 나침반” 경험을 만든다.

## 4. 목표 정보구조

공개 1차 메뉴는 최대 5개로 줄인다.

| 메뉴 | 사용자가 얻는 답 | 기본 화면 |
| --- | --- | --- |
| 오늘의 금천 | 지금 확인할 변화·주의·생활 상태 | 상태 요약, 변화, 빠른 행동 |
| 내 주변 | 현재 위치·주소 주변에 무엇이 있는가 | 지도와 동기화된 목록 |
| 우리 동 | 행정동별 생활 여건은 어떻게 다른가 | 비교 지표, 추세, 설명 |
| 분야별 | 복지·보건·교통·안전 등 깊이 보기 | 분야별 핵심 지표와 서비스 |
| 데이터 찾기 | 원자료·출처·API·다운로드 찾기 | 검색 중심 카탈로그 |

`API 상태`, `API 로그`, `관리자`는 공개 내비게이션에서 제거하고 인증된 운영 콘솔로 분리한다. `이용안내`, 데이터 출처, 접근성 안내는 유틸리티 영역에 둔다. 상권분석은 “분야별 > 지역경제·상권”으로 편입하되 소상공인 수요가 확인되면 별도 진입점을 제공한다.

## 6. 단계별 로드맵

팀 가정은 프런트엔드 2명, 백엔드/데이터 1명, UX 0.5명, QA 0.5명, 의사결정권자 1명이다. 1인 개발이면 병렬 작업을 직렬화해 18~24주로 본다.

### 0단계: 기준선 고정과 안전망, 1주

- 모든 공개·관리 라우트의 데스크톱/태블릿/모바일 스크린샷 기준선 생성
- 브라우저 콘솔·네트워크·키보드 탐색·접근성 자동 검사 추가
- 버그를 P0~P3로 분류하고 신규 기능과 분리
- Mock/실데이터/오류 상태의 표시 규칙 확정

완료 게이트: 전체 라우트 스모크 테스트와 버그 목록 없이 화면 재설계를 시작하지 않는다.

### 1단계: UX 구조와 디자인 방향, 2주

- 주민 핵심 과업 5개와 관리자 과업을 분리
- 공개 메뉴 5개 이내 사이트맵과 화면 흐름 확정
- 홈·내 주변·우리 동 3개 화면의 저해상도/고해상도 프로토타입
- 금천 고유 시각언어, 반응형 그리드, 상태 색상, 타이포 체계 확정

완료 게이트: 주민 5명 이상 과업 테스트에서 핵심 정보 탐색 성공률 80% 이상.

### 2단계: 프런트 안정화와 공통 기반, 3주

- 라우터 생명주기, 비동기 요청, 지도·차트·타이머 정리
- 공통 로딩/빈 값/오류/오래된 데이터/Mock 컴포넌트 구축
- 운영 메뉴 분리, 환경설정과 API 기반 URL 정리
- 큰 페이지 모듈을 기능 단위로 점진 분리
- E2E·접근성·시각 회귀를 CI에 연결

완료 게이트: P0/P1 공개 화면 버그 0건, 핵심 E2E 100% 통과.

### 3단계: 생활 데이터 1차 확대, 3주

- 현재 10개 수집 루틴의 실호출·저장·최신성 검증
- 금천구 고가치 정적/수시 데이터 5~7종 연계
- 내 주변 통합 검색과 시설 상세 카드 구현
- 카탈로그에 출처·필드·갱신주기·품질 상태 공개

완료 게이트: 각 공개 데이터의 원천 건수와 저장 건수 대사, 24시간 이상 오래된 데이터 경고 검증.

### 4단계: 실시간·동별 분석과 시각화, 3주

- 실시간 도시·대기·날씨·교통의 금천구 적용 가능 범위 검증 및 연계
- 동별 인구·생활시설·안전·복지 비교
- 추세, 비교, 분포, 지도 레이어의 표준 컴포넌트 적용
- CSV/GeoJSON 다운로드와 API 설명 연결

완료 게이트: 카드·차트·표의 필터와 수치가 같은 정의로 대사됨.

### 5단계: 공개 베타와 운영 인수, 2주

- 모바일, 저속망, 지도 타일 실패, 외부 API 장애 시나리오 검증
- WCAG 2.2 AA 핵심 기준과 한국어 키보드·스크린리더 점검
- 성능 예산, 관측성, 장애 대응, 데이터 담당자 인수
- 주민 피드백 채널과 개선 주기 운영

완료 게이트: 출시 체크리스트 승인, 롤백 절차와 데이터 중단 공지 절차 검증.

## 8. 성공 지표

### 주민 경험

- 핵심 과업 성공률 85% 이상
- “기준일과 출처를 찾을 수 있음” 테스트 성공률 90% 이상
- 첫 화면에서 주요 생활 상태 확인까지 중앙값 20초 이하
- 모바일 핵심 과업 성공률이 데스크톱 대비 10%p 이상 낮지 않음

### 품질과 안정성

- 공개 P0/P1 오류 0건
- 핵심 경로 E2E 통과율 100%
- 자동 접근성 검사 중대 오류 0건
- 데이터 최신성 SLA 충족률 95% 이상
- 외부 API 실패가 주민 화면의 허위 0건으로 표시되는 비율 0%

### 성능

- 4G 중급 모바일 기준 LCP 2.5초 이하, CLS 0.1 이하를 목표로 측정
- 핵심 화면의 지도 제외 초기 전송량과 실행 시간을 예산으로 관리
- 라우트 전환 후 콘솔 오류와 해제되지 않은 타이머·지도 인스턴스 0건

## 9. 의사결정과 변경 통제

- 매 단계 시작 시 사용자 과업, 데이터 정의, 완료 기준을 먼저 승인한다.
- 신규 데이터 연계와 화면 개편을 같은 PR에 섞지 않는다.
- 수치 정의가 바뀌면 데이터 계약, 카탈로그, 차트 설명, 테스트를 함께 갱신한다.
- 실제 데이터가 없는 화면은 명시적 프로토타입으로만 운영하며 `LIVE` 표현을 사용하지 않는다.
- 기존 `docs/project-roadmap.md`, `docs/code-review-and-improvements.md`, `docs/개선사항(Fable5).md`의 기술 부채 항목은 폐기하지 않고 R2/R5 백로그에서 재검증한다.

## 10. 바로 시작할 첫 스프린트

1. 전체 라우트·상태 조합 테스트 매트릭스를 작성한다.
2. 공개 내비게이션에서 운영 기능을 분리하는 IA 초안을 만든다.
3. 홈 화면의 데이터 상태 표기를 `실데이터/샘플/지연/오류`로 통일한다.
4. 지도 타일 실패 시 목록을 유지하는 회귀 테스트를 만든다.
5. 데이터 후보 11종에 대해 데이터 계약 초안을 작성하고 1차 연계 5종을 확정한다.
6. 홈·내 주변·우리 동 와이어프레임을 주민 과업으로 검증한다.

첫 스프린트 종료 조건은 “새 화면이 예뻐짐”이 아니라 기준선·우선순위·자동 검증·데이터 계약이 승인되는 것이다.
